  • im Dashboard den Sprint Burndown sehen,
  • im Dashboard sehen, wer gerade an was arbeitet,
  • im Dashboard sehen, welche Tasks als nächstes an der Reihe sind (anhand der Priorität),

  • im Sprint Backlog auf einen Blick erkennen, wieviel Prozent von Requirements bereits erledigt sind und wieviele Stunden noch ausstehen,
  • im Sprint Backlog nach dem Öffnen dem expandieren der Requirements, ob Tasks bereits erledigt sind bzw. ob jemand oder wer sie gerade berarbeitet,
  • im Product Backlog auf einen Blick erkennen, wieviele Requirements bereits vom Team geschätzt wurden.
 Der Scrum Master kann
  • im Dashboard anhand eines Balkendiagramms sehen, wieviele Tickets existieren, wieviele noch zu tun und wieviele erledigt sind,

  • Task besteht aus Title, Content, Estimate (hours) und Owners.
  
Übernimmt und schließt Tasks Das Team
  • wählt Whiteboard oder Sprint Backlog aus der Navigation (es öffnet sich das Whiteboard bzw. der Sprint Backlog),
  • (im Sprint Backlog) klickt auf ein Requirement, um es zu expandieren (es werden die Eigenschaften und alle Tasks des Requirements angezeigt),
  • wählt aus dem Aktionsmenü eines Tasks Claim bzw. Close (der Task wird als von dem Benutzer bearbeitet oder als gelöst markiert).
ODER (mit Whiteboard)
  • wählt Whiteboard aus der Navigation (es öffnet sich das Whiteboard),
  • Zieht einen Task aus der Spalte Free Tasks in die Spalte Claimed Tasks oder Closed Tasks (der Position des Task-Blocks ändert sich und der Task wird als von dem Benutzer bearbeitet oder als gelöst markiert).
 (Tasks übernehmen) Das Team
  • wählt einen Sprint Backlog aus einem Drop Down in der Navigation und klickt auf View (es öffnet sich der Sprint Backlog),
  • klickt auf die Owner-Spalte eines Tasks und tippt den Namen des zuständigen (ggf. seinen eigenen) ein (es wird der zuständige für den Task eingetragen; der Task ist damit noch nicht in progress).
(Tasks übernehmen oder schließen) Das Team
  • wählt einen Sprint Backlog aus einem Drop Down in der Navigation und klickt auf View (es öffnet sich der Sprint Backlog),
  • klickt auf die ID eines Tasks (es öffnet sich die View-Ansicht des Tasks),
  • klickt auf die Registerkarte Edit (es öffnet sich eine Edit-Ansicht),
  • kann in Trac-Manier in einer Radiobox-Gruppe den neuen Status zwischen leave as new, resolve as {fixed, invalid, wontfix, duplicate, worksforme}, reassign to [Textfeld] und accept und gibt ggf. einen Change-Kommentar an,
  • klickt auf Preview oder Submit Changes (es wird eine Vorschau des geänderten Tickets angezeigt oder das Ticket geändert (sofern sich das Teammitglied im zuständigen Team befindet, sonst kommt ein Fehler); der Task ist damit noch nicht in progress).
(mit Whiteboard)
  • wählt Whiteboard aus der Navigation (es öffnet sich das Whiteboard; zu welchem Sprint genau, ist mir unklar),
  • zieht an Tasks zwischen den Spalten Backlog, In Progress und Done (der Task wird visuell verschoben und der Status des Tasks wird geändert),
  • falls beim Verschieben in In Progress noch kein Zuständiger eingetragen ist, kommt ein modaler Dialog mit einer Dropdown aller in Frage kommender Teammitglieder.
 (Tasks übernehmen) Das Team
  • wählt Sprint Backlog aus der Toolbar (es öffnet sich das Whiteboard),
  • klickt auf Take im Kontextmenü eines Tasks (der Task wird dem User zugeordnet, dies ist dann im Task im Whiteboard sichtbar).
  • kann Tasks nicht abgeben
(Tasks schließen) Das Team
  • wählt Sprint Backlog aus der Toolbar (es öffnet sich das Whiteboard),
  • draggt ein Task auf die Spalte Locked oder Done
